</div>                         <div class="post_content">               <p><a href="http://cdn.androidpolice.com/wp-content/uploads/2014/11/nexus2cee_2014-11-04-14.45.09.jpg"></a></p> <p>Over the last few years, few topics have been more hotly contested by Android users and developers than how SD cards are handled by the OS. Back in February, I <a href="http://www.androidpolice.com/2014/02/17/external-blues-google-has-brought-big-changes-to-sd-cards-in-kitkat-and-even-samsung-may-be-implementing-them/">discussed some of Google's changes</a> during the transition from Android 2.3 to 4.0, and then how more recent policy changes ultimately led to 3rd-party applications losing most of their access to removable storage. By the time I/O came around, Google acknowledged that KitKat's newly added Storage Access Framework still didn't offer enough range for apps to get their work done. With the release of the L Developer Preview, <a href="http://www.androidpolice.com/2014/06/26/android-l-feature-spotlight-storage-access-framework-enables-batch-operations-full-directory-access/">new APIs were added</a> to allow apps to request access to directories owned by other providers. Now that Android 5.0 Lollipop has been finalized, these APIs have been improved and even offer more capabilities than before, and they do it in a very user-friendly and secure way.</p> <div class="fancy_blockquote"><blockquote><div class="quote_left"></div><p><a href="https://plus.google.com/+JeffSharkey/posts/4kxjY9JFemL" class="external" rel="nofollow" target="_blank">Posted by Jeff Sharkey</a>:</p> <p><b>Richer access to secondary shared storage devices</b></p> <p>In KitKat we introduced APIs that let apps read/write file in app-specific directories on secondary storage devices, such as SD cards.</p> <p>We heard loud and clear that developers wanted richer access beyond these directories, so in Lollipop we added the new ACTION_OPEN_DOCUMENT_TREE intent.&nbsp; Apps can launch this intent to pick and return a directory from any supported DocumentProvider, including any of the shared storage supported by the device.&nbsp; Apps can then create, update, and delete files and directories anywhere under the picked tree without any additional user interaction.&nbsp; Just like the other document intents, apps can persist this access across reboots.</p> <p>This gives apps broad, powerful access to manage files while still involving the user in the initial selection process.&nbsp; Users may choose to give your app access to a narrow directory like &ldquo;My Vacation Photos,&rdquo; or they could pick the top-level of an entire SD card; the choice is theirs.</p> <div class="quote_right"></div></blockquote></div> <p>There aren't a lot of changes to this functionality since the <a href="http://www.androidpolice.com/2014/06/26/android-l-feature-spotlight-storage-access-framework-enables-batch-operations-full-directory-access/">announcement at Google I/O</a>. The Storage Access Framework (SAF) already gave 3rd-party apps the ability to ask for access to a single file or whole directories (including sub-directories). The important detail is mentioned at the end, which is that this can now be used to give total SD card access to an app. Once the decision has been made, apps won't have to request access ever again, and all future operations can occur seamlessly. This finally puts users in charge of which apps can have access different parts of device storage, and it's basically painless. </p><p>That wasn't the only addition regarding storage. Apps that deal heavily in photos, music, and video are prone to filling up space very quickly, and as a result, they may need to push their content to the SD card. KitKat didn't necessarily handle this very well, because the content still had to be placed inside of private folders and inserted into the MediaStore manually. A new method simplifies the process by generating a folder that is essentially open to the public, while still keeping it tied to the app that created it.<br></p><div class="fancy_blockquote"><blockquote><div class="quote_left"></div> <p><a href="https://plus.google.com/105051985738280261832/posts/5i7tXsExLqx" class="external" rel="nofollow" target="_blank">Posted by Dianne Hackborn</a>:</p> <p>Also worth noting is the new getExternalMediaDirs() method that gives you a place for your own files on any available secondary storage, without needing to request read/write permissions: <a href="https://developer.android.com/reference/android/content/Context.html#getExternalMediaDirs()" class="external" rel="nofollow" target="_blank">https://developer.android.com/reference/android/content/Context.html#getExternalMediaDirs()</a></p> <div class="quote_right"></div></blockquote></div> <p>If an app places a file into a folder returned by getExternalMediaDirs, it is essentially available to any other application to access via the MediaStore service. This will be great for something like a camera app or sound recorder, because now the files they produce will be automatically available to every other app without any hassle. This method will also resolve some complications that resulted from removing and reinserting SD cards. Additionally, since the method of access goes through the MediaStore, apps that would consume these files won't even have to be updated for the latest SAF. Of course, they probably should be, so they can also request files that are only available through that method.</p> <p>All put together, this should be enough to alleviate most of the stress related to SD cards after the release of KitKat. Power users will no longer have to deal with crippled file managers, media apps will have convenient access to everything they should regardless of storage location, and developers won't have to rely on <a href="http://www.androidpolice.com/2014/04/06/external-blues-redux-apps-still-have-a-loophole-for-writing-to-the-sd-card-on-kitkat-but-for-how-long/">messy hacks</a> to work around the restrictions. Of course, these features are still specifically a part of Android 5.0 (and above), so it'll be a while these benefits can truly materialize. At least we can all rest easy knowing that the future of Android does not involve dooming the SD card to obscurity &ndash; it's actually balancing usability and security.</p> <p><em>Source: </em><a
